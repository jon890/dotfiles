# 스킬 작성 규칙 (프로젝트 공통)

스킬을 새로 만들거나 수정할 때 항상 적용하는 구조·문체 원칙이다.
skill-creator 의 공식 지침을 일상 작성에서도 따르기 위해 핵심을 추려 두었다.
스킬을 모아 둔 저장소에 자체 규칙(버전·CHANGELOG·validate)이 있으면 그쪽이 우선하고, 본 규칙은 그 위에 더해진다.

## skill-creator 를 활용한다

스킬을 **새로 생성**하거나 **품질을 검증**할 때는 `/skill-creator:skill-creator` 를 호출해 그 워크플로우를 따른다.

- 신규 생성: 의도 파악 → 초안 → 테스트 프롬프트 실행 → 평가 → 개선 반복의 전체 루프를 skill-creator 가 안내한다.
- 기존 스킬 검증·개선: 테스트 실행과 baseline 비교, description 트리거 최적화도 skill-creator 의 기능이다.
- 구조·문체 판단이 애매할 때도 skill-creator 지침을 기준으로 삼는다 (본 rules 는 그 지침의 요약이다).
- 예외: 오타 수정·한 줄 보완 같은 사소한 갱신은 직접 진행해도 된다.

## 디렉터리 구조

```
skill-name/
├── SKILL.md        # 필수 — frontmatter + 본문
├── scripts/        # 반복 실행 코드 (결정적 작업)
├── templates/      # 출력 템플릿 (HTML, 문서 양식 등)
├── references/     # 필요할 때만 로드하는 참조 문서
└── examples/       # 입출력 예시
```

## 반복 레시피는 scripts/ 로 분리한다

SKILL.md 본문에 여러 줄짜리 inline 코드 레시피(heredoc python·복잡한 셸 조립)를 두지 않는다.

이유:

- inline 레시피는 모델이 매 실행마다 코드를 다시 옮겨 적는 구조라 escape 실수·오타가 끼어든다.
- 스크립트 파일은 컨텍스트에 로드되지 않고 바로 실행되므로 토큰·정확성 양쪽에서 유리하다.
- 한 번 작성한 스크립트는 인자만 바꿔 재사용된다 (예: 미리보기 HTML 을 생성하는 `scripts/generate_preview.py` 같은 형태).

판단 기준 — 다음 중 하나면 분리한다:

- 코드가 5줄을 넘는다.
- escape·quoting 이 필요한 heredoc 이다.
- 매 실행 같은 로직을 반복한다.

1~2줄짜리 단순 명령(`gh pr list ...` 같은 한 줄 CLI 호출)은 본문에 그대로 둔다.

## Progressive disclosure — 3단계 로딩

스킬은 3단계로 로드된다는 점을 구조에 반영한다:

1. **name + description** — 항상 컨텍스트에 있음 (트리거 판단 전용)
2. **SKILL.md 본문** — 트리거 시 로드. 500줄 이내 유지.
3. **번들 리소스** — 필요할 때만 로드. scripts/ 는 로드 없이 실행 가능.

- SKILL.md 가 500줄에 다가가면 references/ 로 계층을 나누고 "언제 읽으라"는 안내를 남긴다.
- 300줄 넘는 참조 문서에는 목차를 단다.
- 여러 변형(framework·도메인)을 지원하면 변형별 references 파일로 나눠 해당 파일만 읽게 한다.

## description 작성

description 은 스킬 트리거의 단일 메커니즘이다 — "언제 쓰는지"는 전부 여기에, 본문에는 두지 않는다.

- 트리거 키워드를 한국어·영어 모두 명시한다.
- 모델은 스킬을 덜 쓰는(undertrigger) 경향이 있으므로 다소 적극적으로 쓴다 — "~를 언급하면, 명시적으로 X 라고 하지 않아도 반드시 이 스킬 사용".
- 비슷한 스킬이 있으면 경계를 명시한다 (예: "단순 조회는 조회 전용 스킬 사용").

## 본문 문체

- 명령형으로 쓴다.
- MUST·ALWAYS 남발 대신 **왜 중요한지 이유를 설명한다** — 이유를 알면 모델이 변형 상황에서도 올바르게 행동한다.
- 특정 예시에 과적합하지 않는다 — 함께 다듬은 몇 개 사례에만 맞춘 좁은 규칙 대신 일반 원칙으로 쓴다.
- 실측으로 확인한 함정·키 인사이트는 본문에 남긴다 (예: "이 폼은 fill+Enter 가 안 되고 JS 콜백 직접 호출이 필요").

## 개인 종속 값 외부화

작성자명·계정·경로처럼 사람마다 다른 값은 SKILL.md 에 하드코딩하지 않고 설정 파일(`~/.claude/<skill>.config.json` 등)로 뺀다.
최초 1회 수집 → 저장 → 재사용 흐름을 본문에 명시한다.
이렇게 해야 팀원 누구나 같은 스킬을 본인 환경으로 쓸 수 있다.

## 변경 검증

- 스크립트 분리·리팩토링 시 산출물 동등성을 확인한다 (예: 변경 전후 출력 `cmp` 비교).
- 새 스크립트는 실제 데이터로 1회 실행해 보고 커밋한다.
- 팀 저장소 스킬이면 저장소 검증(`validate.sh` 등)과 버전·CHANGELOG 규칙을 따른다.
