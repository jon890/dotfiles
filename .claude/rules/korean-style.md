# 한국어 표현 정책 (프로젝트 공통)

한국 개발자에게 한 번에 의미가 안 와닿는 외래어·전문용어는 한국어로 풀어 쓴다.
사용자 응답·docs·skill 작성 모두에 적용한다.
코드 식별자(폴더명, 슬래시 커맨드, 코드 심볼, 경로)는 그대로 둔다 — 한국어 prose 만 정리한다.

프로젝트 고유 매핑·예외는 각 프로젝트의 `korean-style.md` 가 본 규칙에 더한다 (있을 때만).

## 외래어 매핑 표

| 금지 | 권장 |
|---|---|
| 매트릭스 (matrix) | 표, 영향 표, 변경 표 |
| 트리아지 (triage) | 분류, 분류 작업 |
| 베이스라인 / baseline | 기준값 (단일 측정값), 기준점 (시점 기준), 기준선 (성능 추이·차트) |
| 스파이크 (spike) | 사전 조사, 탐색 작업, 실증 |
| 게이트 (gate) | 점검, 사전 점검, 통과 조건 |
| wall-time / wall clock time | 전체 처리 시간 (시작-끝), 실제 처리 시간 (CPU time 과 구분 필요 시) |
| silent failure | 묻혀버린 실패, 노출 안 되는 실패 |
| in-flight (요청) | 처리 중인 (요청), 진행 중인 (요청) |
| 리포지토리 위생 (repo hygiene) | 리포지토리 정리, gitignore 정비 |
| 카나리 (canary) | 인스턴스 배포 후 검증 |
| byte-level diff | 바이트 단위 비교 |
| ramp | 단계적 상향 |
| flat fan-out / fan-out | 통합 병렬 호출 / 병렬 호출 |
| ephemeral (instance / runner) | 일회성 (인스턴스 / 러너), 쓰고 버리는 |
| 클램프 / clamp | 값을 범위로 제한, 하한/상한으로 가두기 (예: `max(1, n)` → "1 이상으로 제한") |
| 폭주 (CPU 폭주 등) | 과점유, 과도한 점유 |
| 강등 (로그 레벨 강등) | 낮춤 (예: ERROR → WARNING 으로 낮춤 — 등급제가 아니므로 '강등'은 부적절) |
| 오살 (stale PID 오살) | 풀어 쓰기 (예: "엉뚱한 프로세스를 죽이는 문제") |
| 외과적 (surgical) | 정밀한, 국소적, 범위를 좁힌 (예: "외과적 변경" → "꼭 필요한 곳만 고치는 변경") |

위에 없는 외래어·전문용어도 같은 원칙으로 한국어로 바꾼다.
판단 애매하면 사용자에게 묻는다.

## 쉽고 사실적인 표현 원칙

매핑 표에 없어도 아래 4가지는 항상 점검한다. 멋부린 표현보다 한 번에 읽히는 직설 표현을 쓴다.

1. **생소한 한자어·전문 조어 회피** — 사전에 없거나 한 번에 안 읽히는 압축 한자어는 풀어 쓴다.
   - 예: "오살" → "엉뚱한 프로세스를 죽이는 문제"
2. **과장·구어 표현 회피** — 중립적 기술 표현을 쓴다.
   - 예: "CPU 폭주" → "CPU 과점유"
3. **맥락에 안 맞는 비유어 회피** — 직설 동사로 바꾼다.
   - 예: "로그 레벨 강등" → "로그 레벨을 낮춤" (로그 레벨은 등급제가 아님)
4. **비유·외래어 대신 실제 동작을 그대로 서술** — 무엇을 하는지 그대로 적는다.
   - 예: "응답을 녹화·재생" / "mock 환경" → "응답 값을 미리 저장해 두고 그대로 사용"

## 영문 관용 유지

한국 개발자도 영문에 더 익숙한 용어는 그대로 쓴다:

- `rate limit` / `autoscaling` / `downstream` / `pipeline` / `backpressure`
- 코드·도구 식별자 (`yield` / `tuple` / `Counter` / `Release` 등)
- 언어 관용 (generator / context manager 등)

## 영어 표기 유지와 최초 병기

다음 관측성·운영 용어는 한글로 풀지 않고 **영어 표준 표기를 유지**하되, 기술 개념·블로그 문서에서 **처음 등장할 때만 한국어를 병기**한다 — `Latency`(지연) 형태. 영어를 빼면 검색·인터뷰·실무 표기와 어긋나기 때문이다.

- `Latency`(지연), `Throughput`(처리량), `Saturation`(포화도)

맥락에 따라 적용을 가른다:

- 독자가 개발자인 기술 개념·블로그 문서 → 영어 표기 유지, 최초 1회 병기.
- 업무 보고·일반 설명·채팅 응답 → 한국어로 풀어 설명한다.

## 기존 docs 에서 발견 시

- 작업 중인 파일: 함께 정리한다.
- 작업 외 파일: 사용자에게 알리고 별도 정리 여부를 확인한다.

## 문장 종결 규칙

- **명사형 종결 회피 (본문 문장)** — paragraph 의 평문 문장은 동사로 끝맺어 의미를 완결한다.
  - 금지: "측정 필요.", "정확도 회귀 0건 확보.", "동작 불변."
  - 권장: "측정한다.", "확보하는 것이 목표이다.", "동작이 변하지 않는다."
  - 예외 — bullet list / 표 셀 / 헤더 / 부가 설명 안의 항목은 명사구 종결 OK.

## 자가 점검

문서 작성 직후 아래 항목을 확인한다:

1. 매핑 표 단어가 잔존하는가? → 한국어로 풀이한다.
2. paragraph 평문 문장이 명사형으로 끝나는가? → 동사로 풀이한다.
3. 전문 용어가 첫 등장 시 풀이 없이 쓰였는가? → 풀이를 추가한다.
4. 생소한 한자어·과장어·맥락 안 맞는 비유어가 있는가? → 쉬운 직설 표현으로 바꾼다.
5. 동작을 비유(녹화·재생 등)로 표현했는가? → 실제로 무엇을 하는지 서술로 바꾼다.
